# 全栈项目代码注释与文档规范 (v1.0)

## 0. 引言
良好的注释不仅能降低团队沟通成本，还能提升代码可维护性。本项目采用 React + FastAPI 架构，由于 FastAPI 具有“代码即文档”的特性，后端的注释直接决定了接口文档的质量，进而影响前端的开发效率。因此，特制定本规范。

## 1. 基本原则
1. **中英文结合**：代码的命名（变量、函数、类）**必须**使用具有明确语义的英文；注释**推荐**使用中文，以降低阅读理解成本。
2. **解释“为什么”而不是“是什么”**：代码本身应能表达“做了什么”（What），注释应重点解释“为什么这么做”（Why），特别是复杂的业务逻辑、算法或绕过特定 Bug 的处理。
3. **注释与代码同步更新**：修改代码时，**必须**同步修改相关注释。误导性的注释比没有注释更糟糕。
4. **留白与排版**：注释与上方代码块之间应保留一个空行，以提升可读性。

## 2. 全局特殊标记规范
全栈代码（前后端）统一使用以下关键字进行特殊标记。格式要求：`关键字(责任人): 说明 [关联Issue/日期]`。

* **`TODO`**：待办事项。表示该功能尚未完成或需要进一步完善。
  > `// TODO(张三): 接入真实的第三方支付接口，目前为 Mock 数据。 #ISSUE-10`
* **`FIXME`**：待修复。表示已知此处有 Bug 或逻辑缺陷，但暂时不影响主流程，需尽快修复。
  > `# FIXME(李四): 当用户并发点击时可能会产生重复订单，需要加锁。`
* **`HACK`**：临时性妥协。表示由于时间或技术限制，采用了不优雅的、暴力的手段解决问题，提醒后续重构。
  > `// HACK(王五): 强行设置 setTimeout 200ms 等待动画结束，下个版本用 TransitionEnd 事件重构。`
* **`NOTE`**：重点说明。提醒阅读者注意某个复杂的机制或容易踩坑的地方。

*(建议搭配 VSCode 插件 `Todo Tree` 使用，以便在侧边栏集中管理这些标记。)*

## 3. 后端注释规范 (Python + FastAPI)

后端代码严格遵守 **PEP 8** 规范，Docstring 采用 **Google Style** 格式。

### 3.1 路由与接口注释（核心）
FastAPI 的路由装饰器和函数的 Docstring 会直接生成 Swagger UI 文档，必须极其严谨。

* `summary`：接口的一句话简述（显示为 Swagger 标题）。
* `description`：接口详细业务逻辑说明。
* `Docstring`：详细阐述参数、返回值及可能抛出的异常。

**示例：**
```python
@router.post(
    "/users",
    summary="创建新用户",
    description="接收前端传递的用户信息，校验后存入数据库，并触发欢迎邮件发送任务。",
    response_description="返回创建成功的用户 ID",
    tags=["Users"]
)
async def create_user(user_in: UserCreate) -> dict:
    """
    执行创建用户的核心逻辑。
    
    Args:
        user_in (UserCreate): 包含邮箱和密码的 Pydantic 模型

    Returns:
        dict: 包含新用户 UUID 的字典
        
    Raises:
        HTTPException(400): 如果邮箱已被注册
        HTTPException(500): 数据库写入失败
    """
    pass
```

### 3.2 数据模型注释 (Pydantic / SQLAlchemy)
Pydantic 模型的 `Docstring` 和字段的 `Field` 会直接反映在 Swagger 的 Schemas 中。

**示例：**
```python
class UserCreate(BaseModel):
    """用户注册请求模型"""
    
    email: EmailStr = Field(..., title="用户邮箱", description="必须是有效的企业邮箱格式")
    password: str = Field(..., min_length=8, description="密码长度至少为8位，需包含字母和数字")
```

## 4. 前端注释规范 (React + TS/JS)

前端代码推荐使用 **TSDoc (或 JSDoc)** 标准。利用 TypeScript 的类型系统，让 IDE 自动推导和提示注释。

### 4.1 组件与 Props 注释
组件必须包含功能说明，Props 接口的每个非显而易见的属性都必须有注释。

**示例：**
```tsx
/**
 * 带有权限校验的按钮组件
 * 
 * 只有当当前用户拥有指定 action 权限时才会渲染，否则隐藏或禁用。
 * 
 * @example
 * <AuthButton action="delete_user" onClick={handleDelete}>删除用户</AuthButton>
 */
export const AuthButton: React.FC<AuthButtonProps> = ({ action, children, ...props }) => {
    // ...
}

interface AuthButtonProps extends React.ButtonHTMLAttributes<HTMLButtonElement> {
    /** 
     * 需要校验的权限标识符
     * @see permissions.ts 中的权限常量表
     */
    action: string;
    /** 如果无权限，是隐藏按钮还是仅禁用？ @default 'hide' */
    fallbackMode?: 'hide' | 'disable';
}
```

### 4.2 自定义 Hook 注释
Hook 必须清晰地说明入参和返回值。

**示例：**
```typescript
/**
 * 分页数据请求 Hook
 * 
 * 封装了针对列表型 API 的页码控制、加载状态和数据存储逻辑。
 * 
 * @param apiEndpoint 请求的 API 路径
 * @param pageSize 每页条数，默认为 10
 * @returns 包含 data, loading, error, 和 分页操作函数的对象
 */
export const usePagination = <T>(apiEndpoint: string, pageSize = 10) => {
   // ...
}
```

### 4.3 复杂逻辑的行内注释
使用单行注释 `//` 解释复杂逻辑段落的**目的**。

```typescript
// 将后端返回的一维平铺菜单数据，转换为 UI 组件需要的树形结构
const treeData = buildTree(flatData, rootId);
```

## 5. Git 提交规范 (Conventional Commits)

代码的提交历史是项目最重要的文档之一。提交信息格式必须符合以下标准：

`提交类型(作用域): 简短说明`

**常用的提交类型（Type）：**
* `feat`: 新增功能 (Feature)
* `fix`: 修复 Bug
* `docs`: 仅修改文档/注释
* `style`: 代码格式修改（不影响逻辑，如空格、缩进、分号等）
* `refactor`: 重构（既不是新增功能，也不是修复 Bug 的代码变动）
* `perf`: 优化性能
* `chore`: 构建过程或辅助工具的变动（如更新 npm 包，修改 Dockerfile）

**示例：**
* `feat(auth): 增加微信扫码登录接口`
* `fix(table): 修复表格在移动端宽度溢出的问题`
* `docs(api): 更新用户模块的 Swagger 描述`

## 6. 工具与强制执行 (IDE & Linting)

规范的落地依赖于工具的辅助：

1. **VSCode 必备插件推荐**：
   * `autoDocstring` (后端): Python Docstring 自动生成工具（设置格式为 Google）。
   * `Todo Tree` (前后端通用): 高亮并树状展示 TODO、FIXME 等标记。
   * `Error Lens` (前后端通用): 直接在代码行内显示 Linter 报错。

2. **代码检查（CI/CD 或 Git Hooks）**：
   * **前端**：采用 `Husky` + `Commitlint` 拦截不符合规范的 Git 提交；使用 `ESLint` (`eslint-plugin-jsdoc`) 检查缺失的注释。
   * **后端**：使用 `Ruff` 或 `Flake8` (`flake8-docstrings`) 在代码提交前检查是否按规范编写了 Docstring。
