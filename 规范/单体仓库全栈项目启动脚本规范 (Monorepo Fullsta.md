# 单体仓库全栈项目启动脚本规范 (Monorepo Fullstack Startup Script Specification)

## 1. 概述

在单体仓库（Monorepo）结构的全栈项目中，通常包含前端应用、后端 API 服务、后台任务队列（Worker）等多个独立运行的子系统。为了确保项目在不同环境（开发、测试、生产）和不同操作系统（Windows、Linux、macOS）下均能稳定、可靠地启动，并遵循"高内聚、低耦合"的架构原则，特制定本启动脚本规范。

本规范旨在解决多进程管理、跨平台兼容、日志混乱及环境依赖等痛点，为全栈项目提供一个健壮的统一启动控制台方案。

## 2. 核心原则

1.  **高内聚，低耦合**：主启动脚本（如 `start.py` 或 `start.sh`）仅负责调度、进程管理和用户交互；各子服务（如 API、Worker、Web）的启动逻辑应封装在独立的脚本中。
2.  **防御性编程**：在真正拉起服务进程前，必须进行充分的前置检查（Pre-flight Checks），避免服务在启动中途因环境问题崩溃。
3.  **优雅降级与退出**：必须妥善处理进程树的生命周期，确保在接收到中断信号时，所有子进程及孙进程都能被安全、彻底地清理，不留僵尸进程。
4.  **统一可观测性**：规范化日志输出与控制台视觉呈现，确保多进程并发启动时信息清晰可读。

## 3. 规范细则

### 3.1 环境与依赖的前置检查 (Pre-flight Checks)

在启动任何核心服务之前，启动脚本必须执行以下检查：

*   **配置文件检查**：验证环境变量文件（如 `.env`）是否存在。若不存在，应提示用户从模板文件（如 `.env.example`）复制并配置。
*   **关键依赖检查**：检查执行环境（如 Python 虚拟环境、Node.js 包管理器等）是否就绪。
*   **端口占用检查**：在启动需要绑定端口的服务（如 API 服务、前端开发服务器）前，必须检查目标端口是否已被占用。若被占用，应立即报错并终止该服务的启动流程。
*   **基础设施连通性检查**：对于强依赖外部组件的服务（如依赖 Redis 的任务队列，依赖数据库的 API），应在启动前尝试建立短暂的探测连接（Ping）。若连接失败，应提供明确的错误提示（如"数据库连接失败，请检查配置或确认服务已启动"）。

### 3.2 进程管理的健壮性 (Process Management)

启动脚本必须能够安全地管理整个进程树，防止产生孤儿进程或僵尸进程。

*   **进程组管理 (Linux/macOS)**：在类 Unix 系统下，启动子进程时应将其设置为新的进程组组长（例如在 Python 中使用 `os.setsid`）。在清理时，向整个进程组发送信号（如 `os.killpg`），以确保由子进程派生出的孙进程（如带有热重载功能的服务进程）也能被一并终止。
*   **进程树终止 (Windows)**：在 Windows 系统下，由于缺乏 POSIX 信号机制，应使用系统命令（如 `taskkill /F /T /PID <pid>`）来递归终止整个进程树。
*   **状态监控**：主脚本应具备基本的守护（Supervisor）能力，能够捕获子进程的意外退出，并给出相应的日志提示。

### 3.3 日志的统一管理与隔离 (Logging)

为了解决多进程日志交错导致的可读性问题，必须规范日志的输出范式。

*   **统一日志范式**：**若项目中已存在统一的日志模块，则必须优先复用该模块进行日志记录，避免重复造轮子。**
*   **日志前缀与颜色区分**：在控制台输出时，应为不同服务的日志添加带有颜色的明确前缀（例如 `[API]`, `[Worker]`, `[Web]`），以便开发者快速区分日志来源。
*   **日志重定向与持久化**：除了控制台的标准输出，建议将各子服务的标准输出和标准错误重定向到独立的日志文件中（例如 `logs/api.log`, `logs/worker.log`），以便于事后排查复杂问题。

### 3.4 跨平台兼容性 (Cross-Platform Compatibility)

启动脚本必须屏蔽底层操作系统的差异，保持业务逻辑的纯粹性。

*   **平台差异封装**：**严禁在业务启动逻辑中直接硬编码平台判断语句（如 `if sys.platform == 'win32'`）。**
*   **统一工具模块**：所有的平台差异（如信号处理、进程树清理策略、路径拼接规则）必须被抽象并封装到一个独立的工具模块中（例如 `scripts/utils/process_utils.py`）。
*   **接口调用**：各子启动脚本只需调用该工具模块提供的统一接口（如 `graceful_terminate(process)`），由工具模块在内部处理不同操作系统的具体实现。

### 3.5 命令行参数支持 (CLI Arguments)

为了兼顾本地开发的便利性和自动化部署的需求，启动脚本应支持多种交互模式。

*   **交互式菜单**：在无参数启动时，提供友好的交互式菜单，供开发者手动选择需要启动的服务组合。
*   **命令行参数**：引入参数解析库（如 Python 的 `argparse` 或 `click`），支持通过命令行参数直接指定启动行为（例如 `--services api,worker`）。这对于 CI/CD 流程、Docker 容器的 Entrypoint 或自动化测试脚本至关重要。

### 3.6 控制台视觉呈现 (Console UI)

控制台是开发者与启动脚本交互的唯一界面，清晰、一致的视觉呈现能够显著降低认知负担。本节规范控制台输出的结构、颜色和信息密度。

#### 3.6.1 颜色与语义绑定

颜色应与信息语义严格绑定，**不得将颜色用于纯装饰目的**。推荐使用如下语义映射：

| 颜色 | 语义 | 适用场景 |
|---|---|---|
| **绿色** | 成功 / 就绪 | 服务启动成功、前置检查通过 |
| **黄色** | 警告 / 等待 | 服务启动中、等待连接、非致命异常 |
| **红色** | 错误 / 失败 | 前置检查失败、服务崩溃、端口冲突 |
| **青色 / 蓝色** | 信息 / 提示 | 菜单选项、操作引导、PID 等元数据 |
| **白色 / 默认** | 普通输出 | 服务的标准日志流 |

颜色输出应通过统一的工具函数（如 `log_utils.py` 中的 `print_colored`）实现，**严禁在业务脚本中直接硬编码 ANSI 转义码**。此外，必须检测当前终端是否支持颜色输出（如 `sys.stdout.isatty()`），在不支持的环境（如日志文件重定向、CI 管道）中自动降级为纯文本输出。

#### 3.6.2 布局结构

控制台输出应遵循以下固定的生命周期布局，使开发者能够一眼判断当前处于哪个阶段：

```
╔══════════════════════════════════════════════════════════════════════╗
║                    [项目名称] 服务启动控制台                          ║
╚══════════════════════════════════════════════════════════════════════╝

[阶段一：前置检查]
  ✔  .env 配置文件        已就绪
  ✔  数据库连接           已就绪
  ✔  Redis 连接           已就绪
  ✔  端口 8000            未被占用
  ✔  端口 5173            未被占用

──────────────────────────────────────────────────────────────────────

[阶段二：服务启动]
  ● API 服务              正在启动... (PID: 12345)
  ● Worker 服务           正在启动... (PID: 12346)
  ● Web 服务              正在启动... (PID: 12347)

──────────────────────────────────────────────────────────────────────

[阶段三：运行中]
  所有服务已就绪。按下 Ctrl+C 可安全终止所有服务。

[API]    INFO     Application startup complete.
[Worker] INFO     celery@host ready.
[Web]    INFO     Local:   http://localhost:5173/
```

上述布局体现了以下设计原则：
- **阶段分隔**：使用分隔线（`──`）明确区分"检查"、"启动"、"运行"三个阶段，避免日志混淆。
- **对齐输出**：同类信息（如前置检查项、服务名称）应对齐排列，便于快速扫描。
- **状态符号**：使用统一的状态符号（`✔` 成功、`✘` 失败、`●` 进行中）替代纯文字描述，提升信息密度。

#### 3.6.3 前置检查的输出规范

前置检查的输出必须是**逐项实时打印**的，而非等待所有检查完成后一次性输出。每一项检查应包含：检查项名称、检查结果（成功/失败）以及失败时的具体原因和修复建议。

```
  ✔  数据库连接           已就绪
  ✘  Redis 连接           连接失败 → 请确认 Redis 服务已启动，或检查 .env 中的 REDIS_URL 配置
```

若任意**强依赖**的前置检查失败，启动流程必须立即中止，并以非零状态码退出，不得继续尝试拉起服务进程。

#### 3.6.4 服务运行阶段的日志前缀规范

在多个服务并发运行、日志合并输出至同一控制台时，每一行日志必须携带**固定宽度的服务名前缀**，以确保多行日志在视觉上对齐：

```
[API   ]  INFO     Received request: GET /api/health
[Worker]  INFO     Task received: generate_title
[Web   ]  INFO     hmr update /src/App.tsx
```

前缀宽度应以最长的服务名为基准进行右侧空格填充，保证所有服务的日志正文在同一列开始。

#### 3.6.5 退出阶段的输出规范

接收到终止信号后，控制台应清晰展示每个服务的关闭进度，不得在无任何输出的情况下静默退出：

```
──────────────────────────────────────────────────────────────────────
接收到退出信号，正在安全关闭所有服务...

  → 正在终止 API 服务    (PID: 12345) ... ✔ 已关闭
  → 正在终止 Worker 服务 (PID: 12346) ... ✔ 已关闭
  → 正在终止 Web 服务    (PID: 12347) ... ✔ 已关闭

所有服务已安全退出。
──────────────────────────────────────────────────────────────────────
```

若某个服务在超时后仍未退出，应在该行末尾标注"强制终止"，并以警告色输出，以便开发者知晓该服务未能优雅退出。

## 4. 架构设计建议 (目录结构)

为了落实上述规范，建议对单体仓库的 `scripts` 目录进行如下结构优化（以 Python 为例）：

```text
scripts/
├── utils/
│   ├── __init__.py
│   ├── process_utils.py   # 封装跨平台的进程管理、信号处理、优雅退出逻辑
│   ├── check_utils.py     # 封装端口检查、环境检查、连通性探测逻辑
│   └── log_utils.py       # 封装日志前缀、颜色、对齐输出逻辑（基于项目统一日志模块）
├── start.py               # 主入口：解析 CLI 参数，展示菜单，调度子脚本
├── start_api.py           # API 服务启动逻辑（调用 utils 进行前置检查和进程拉起）
├── start_worker.py        # Worker 服务启动逻辑
└── start_web.py           # Web 服务启动逻辑
```
